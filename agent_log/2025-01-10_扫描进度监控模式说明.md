# 扫描进度监控模式说明

## 日期
2025-01-10

## 概述

在媒体服务的文件扫描系统中，有两种主要的文件监控模式：**watchdog模式**和**polling模式**。这两种模式用于监控文件系统的变化，以便及时触发扫描任务。

## 一、Watchdog模式

### 定义
Watchdog模式是一种基于**事件驱动**的文件系统监控方式，使用Python的`watchdog`库来监听文件系统事件。

### 工作原理

1. **事件监听机制**
   - 使用`watchdog.observers.Observer`创建文件系统观察者
   - 为每个监控路径创建`FileSystemEventHandler`事件处理器
   - 监听以下文件事件：
     - `on_created`: 文件创建事件
     - `on_modified`: 文件修改事件
     - `on_moved`: 文件移动事件

2. **事件处理流程**
   ```
   文件系统变化 → Watchdog捕获事件 → 事件处理器 → 防抖处理 → 触发扫描
   ```

3. **防抖机制**
   - 避免短时间内多次触发扫描
   - 使用`debounce_seconds`配置防抖时间（默认5秒）
   - 累积待扫描文件，防抖时间结束后统一扫描

### 代码实现要点

```python
class MediaFileEventHandler(FileSystemEventHandler):
    """媒体文件事件处理器"""

    def on_created(self, event: FileSystemEvent):
        """文件创建事件"""
        if not event.is_directory:
            if self._is_media_file(event.src_path):
                self._debounce_scan(event.src_path)

    def _debounce_scan(self, file_path: str):
        """防抖处理，避免频繁扫描"""
        self.pending_files.add(file_path)
        # 取消之前的定时器，创建新的定时器
        if self.debounce_timer and not self.debounce_timer.done():
            self.debounce_timer.cancel()
        self.debounce_timer = asyncio.create_task(self._delayed_scan())
```

### 优点

1. **实时性强**
   - 文件变化立即被捕获
   - 无延迟触发扫描

2. **资源效率高**
   - 只在文件变化时触发
   - 不需要持续轮询

3. **精确度高**
   - 直接捕获文件系统事件
   - 不会遗漏任何变化

### 缺点

1. **系统依赖性**
   - 依赖操作系统的文件系统通知机制
   - 不同平台实现可能不一致

2. **资源占用**
   - 需要为每个监控路径创建观察者
   - 大量路径可能消耗较多资源

3. **网络文件系统支持**
   - 对网络文件系统（如NFS、SMB）支持有限
   - 某些网络文件系统不支持事件通知

## 二、Polling模式

### 定义
Polling模式是一种基于**定时轮询**的文件系统监控方式，通过定期检查文件系统状态来发现变化。

### 工作原理

1. **轮询机制**
   - 按照固定时间间隔（如每5分钟）检查文件系统
   - 比较当前文件列表与上次扫描结果
   - 发现新增或修改的文件

2. **扫描流程**
   ```
   定时触发 → 扫描文件系统 → 比较文件状态 → 识别变化 → 更新数据库
   ```

3. **状态比较**
   - 比较文件路径、修改时间、文件大小
   - 检测新增、修改、删除的文件

### 代码实现要点

```python
async def polling_scan(paths: List[str], interval: int):
    """定时轮询扫描"""
    while True:
        for path in paths:
            # 扫描文件系统
            scanner = FileScanner()
            await scanner.scan(path)

            # 比较文件状态
            new_files = scanner.new_files
            updated_files = scanner.updated_files

            # 处理变化
            if new_files or updated_files:
                await process_changes(new_files, updated_files)

        # 等待下次轮询
        await asyncio.sleep(interval)
```

### 优点

1. **通用性强**
   - 不依赖操作系统特性
   - 适用于所有文件系统类型
   - 对网络文件系统支持良好

2. **可控性高**
   - 轮询间隔可配置
   - 资源使用可预测

3. **实现简单**
   - 逻辑直观易懂
   - 不需要处理复杂的事件

### 缺点

1. **实时性差**
   - 存在轮询间隔延迟
   - 文件变化不能立即被发现

2. **资源消耗**
   - 持续轮询消耗CPU和IO资源
   - 大量文件时扫描时间长

3. **精确度低**
   - 可能遗漏间隔内的快速变化
   - 无法精确捕获变化时间点

## 三、两种模式的对比

| 特性 | Watchdog模式 | Polling模式 |
|------|-------------|-------------|
| **触发方式** | 事件驱动 | 定时轮询 |
| **实时性** | 高（毫秒级） | 低（取决于轮询间隔） |
| **资源消耗** | 低（事件触发时） | 高（持续轮询） |
| **系统依赖** | 高（依赖文件系统通知） | 低（通用性强） |
| **网络文件系统** | 支持有限 | 支持良好 |
| **实现复杂度** | 中等 | 简单 |
| **精确度** | 高（精确捕获事件） | 低（可能遗漏变化） |
| **适用场景** | 本地文件系统、需要实时响应 | 网络文件系统、资源受限环境 |

## 四、配置说明

在项目的扫描路径配置中，可以通过以下参数配置监控模式：

```python
class ScanPathConfigResponse(BaseModel):
    monitoring_enabled: bool  # 是否启用监控
    monitoring_mode: str     # 监控模式：watchdog/polling
    monitoring_debounce: int  # 监控防抖时间（秒）
```

### 配置示例

#### Watchdog模式配置
```python
{
    "monitoring_enabled": True,
    "monitoring_mode": "watchdog",
    "monitoring_debounce": 5  # 5秒防抖
}
```

#### Polling模式配置
```python
{
    "monitoring_enabled": True,
    "monitoring_mode": "polling",
    "monitoring_debounce": 300  # 5分钟轮询间隔
}
```

## 五、使用建议

### 适合使用Watchdog模式的场景

1. 本地文件系统监控
2. 需要实时响应文件变化
3. 文件变化频率较低
4. 系统资源充足

### 适合使用Polling模式的场景

1. 网络文件系统（NFS、SMB等）
2. 不需要实时响应
3. 系统资源受限
4. 文件变化频繁
5. 需要跨平台兼容

### 混合使用策略

可以根据不同路径的特点，灵活选择监控模式：

```python
paths = [
    {
        "path": "/local/media",
        "monitoring_mode": "watchdog",  # 本地路径使用watchdog
        "monitoring_debounce": 5
    },
    {
        "path": "/nas/media",
        "monitoring_mode": "polling",  # 网络路径使用polling
        "monitoring_debounce": 300
    }
]
```

## 总结

Watchdog模式和Polling模式各有优劣，选择哪种模式取决于具体的使用场景和需求：

- **Watchdog模式**：适合需要实时响应的本地文件系统监控
- **Polling模式**：适合网络文件系统或不需要实时响应的场景

在实际应用中，可以根据不同路径的特点灵活选择监控模式，甚至混合使用两种模式，以达到最佳的性能和用户体验。
