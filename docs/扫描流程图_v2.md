## 一、系统架构概述

### 核心数据模型

#### 1. 扫描任务（批量级别）
**ScanHistory** - 扫描历史记录
- 记录一次扫描任务的总体信息
- 批次ID（batch_id）唯一标识一次扫描
- 包含扫描路径、扫描类型、总体统计等
- 对应前端的"扫描任务"列表

**ScanProgress** - 扫描进度记录
- 记录扫描任务的实时进度
- 关联batch_id和task_id
- 包含已扫描文件数、新增文件数、当前文件等
- 通过WebSocket实时推送到前端

#### 2. 文件任务（文件级别）
**FileTask** - 文件级别任务
- 每个被扫描的文件对应一个FileTask记录
- 记录文件在扫描、识别、整理各阶段的状态
- 支持独立查询每个文件的处理进度
- 关联MediaFile记录

**任务流程**：
```
pending → scanning → scanned → recognizing → recognized → organizing → organized
   ↓         ↓         ↓          ↓          ↓          ↓
 failed     failed    failed     failed      failed      failed
```

```

前端: handleScan()

检查扫描路径配置

   ↓
显示扫描选择对话框
   ↓
用户选择扫描方式

   │  └─ 显示扫描次数

   │  使用路径配置（scan_type, recursive, ignore_patterns等）
   │  ↓

   │

      ├─ 扫描类型（全量/增量）
      ├─ 递归扫描
      ├─ 文件跳过模式（keyword/record/none）
      └─ 忽略文件模式
      ↓
      调用API: POST /api/scan/tasks
         body: {
           path: "<path>",
           recursive: true,
           scan_type: "full",
           skip_mode: "keyword"
         }
   ↓
后端: POST /api/scan/tasks
   ├─ 验证路径是否存在
   ├─ 检查路径是否为目录

   │  ├─ batch_id = batch_id

   │  └─ started_at = 当前时间

   ├─ 统计总文件数 (total_files)

   │  ├─ updated_files = [] (更新文件列表)
   │  ├─ failed_files = [] (失败文件列表)
   │  └─ skipped_files = 0 (跳过计数)
   │  ↓
   │  【遍历文件】
   │  ↓
   │  对每个文件进行处理
   │  ↓
   │  检查停止标志
   │  ├─ 已停止 → 终止扫描
   │  └─ 未停止 → 继续
   │  ↓

   │  ↓

   │  └─ 不是关键词库文件 → 继续
   │  ↓
   │  【处理媒体文件】
   │  ↓
   │  _process_media_file(file_path, scan_type)
   │  ↓
   │  提取文件基本信息
   │  ├─ file_path = 文件绝对路径
   │  ├─ file_name = 文件名
   │  ├─ file_extension = 扩展名
   │  ├─ file_size = 文件大小
   │  ├─ create_time = 创建时间
   │  ├─ modify_time = 修改时间
   │  └─ access_time = 访问时间
   │  ↓
   │  提取媒体元数据
   │  ├─ 使用MediaInfo解析
   │  ├─ 获取视频轨道信息
   │  ├─ 获取音频轨道信息
   │  ├─ 获取字幕轨道信息
   │  └─ 计算视频分辨率、比特率等
   │  ↓
   │  【数据库操作】
   │  ↓
   │  查询文件是否已存在
   │  ↓
   │  创建/更新FileTask记录
   │  ↓
   │  【场景1: 新文件】
   │  ↓
   │  文件不存在 → 创建新记录
   │  ├─ 创建MediaFile记录
   │  │  ├─ 插入文件基本信息
   │  │  ├─ 插入媒体元数据
   │  │  └─ 计算文件哈希
   │  └─ 创建FileTask记录
   │     ├─ media_file_id = MediaFile.id
   │     ├─ scan_batch_id = batch_id
   │     ├─ target_path = 文件父目录
   │     ├─ file_name = 文件名
   │     ├─ status = "scanning"
   │     ├─ scan_started_at = 当前时间
   │     └─ 提交到数据库
   │  ↓
   │  【场景2: 已存在文件】
   │  ↓
   │  文件已存在 → 创建FileTask记录
   │  ├─ media_file_id = existing_file.id
   │  ├─ scan_batch_id = batch_id
   │  ├─ target_path = 文件父目录
   │  ├─ file_name = 文件名
   │  ├─ status = "scanning"
   │  ├─ scan_started_at = 当前时间
   │  └─ 提交到数据库
   │  ↓
   │  【根据scan_type处理】
   │  ↓
   │  └─ scan_type = "full" (全量扫描)
   │     ↓
   │     强制更新文件元数据
   │     ├─ 更新MediaFile记录
   │     │  ├─ 更新文件基本信息
   │     │  ├─ 更新媒体元数据
   │     │  ├─ 更新scanned_at = 当前时间
   │     │  └─ updated_files.append(file_data)
   │     └─ 更新FileTask记录
   │        ├─ status = "scanned"
   │        └─ scan_completed_at = 当前时间
   │
   │  └─ scan_type = "incremental" (增量扫描)
   │     ↓
   │     比较修改时间和文件大小
   │     ├─ 相同 → 跳过文件
   │     │  ├─ skipped_files + 1
   │     │  └─ 更新FileTask为"skipped"
   │     │     ├─ status = "skipped"
   │     │     └─ scan_completed_at = 当前时间
   │     └─ 不同 → 更新文件
   │        ├─ 更新MediaFile记录
   │        │  ├─ 更新文件元数据
   │        │  └─ updated_files.append(file_data)
   │        └─ 更新FileTask记录
   │           ├─ status = "scanned"
   │           └─ scan_completed_at = 当前时间
   │
   │  └─ scan_type = "rescan" (重新扫描)
   │     ↓
   │     强制更新文件元数据
   │     ├─ 重新计算文件哈希
   │     ├─ 更新MediaFile记录
   │     │  ├─ 更新所有字段
   │     │  └─ updated_files.append(file_data)
   │     └─ 更新FileTask记录
   │        ├─ status = "scanned"
   │        └─ scan_completed_at = 当前时间
   │
   │  【更新统计信息】
   │  ↓
   │  ├─ scanned_files.add(file_path)
   │  └─ 更新进度计数
   │     ├─ len(scanned_files)
   │     ├─ len(new_files)
   │     ├─ len(updated_files)
   │     ├─ skipped_files
   │     └─ len(failed_files)
   │  ↓
   │  【处理字幕文件】
   │  ↓
   │  _process_subtitle_file(file_path)
   │  ├─ 查找关联的媒体文件
   │  ├─ 检查字幕是否已存在
   │  └─ 创建SubtitleFile记录
   │
   【进度更新】
   ↓
   每分钟更新进度
   ├─ 检查时间间隔
   │  └─ 距上次更新 < 1分钟 → 跳过
   │
   ├─ 准备进度数据
   │  ├─ batch_id = batch_id
   │  ├─ task_id = task_id
   │  ├─ status = "running" 或 "completed"
   │  ├─ total_files = len(scanned_files)
   │  ├─ scanned_files = len(scanned_files)
   │  ├─ new_files = len(new_files)
   │  ├─ updated_files = len(updated_files)
   │  ├─ skipped_files = skipped_files
   │  ├─ failed_files = len(failed_files)
   │  ├─ current_file = 当前文件路径
   │  └─ progress = (已扫描/总数) * 100
   │
   ├─ 更新数据库 (ScanProgress)
   │  ├─ 查询并更新进度记录
   │  ├─ 更新last_updated_at
   │  └─ 提交事务
   │
   └─ 通过WebSocket推送
      ├─ manager.send_progress(task_id, progress_data)
      └─ 发送到所有活跃的WebSocket连接
   ↓
【扫描完成】
   ↓
更新最终状态
   ├─ 计算扫描耗时 (duration = now - start_time)
   ├─ 确定最终状态 (completed/stopped)
   ├─ 更新ScanHistory记录
   │  ├─ total_files = len(scanned_files)
   │  ├─ new_files = len(new_files)
   │  ├─ updated_files = len(updated_files)
   │  ├─ skipped_files = skipped_files
   │  ├─ failed_files = len(failed_files)
   │  ├─ duration_seconds = duration
   │  ├─ completed_at = 当前时间
   │  └─ status = final_status
   ├─ 更新ScanProgress记录
   │  ├─ status = final_status
   │  ├─ progress = 100
   │  ├─ completed_at = 当前时间
   │  └─ last_updated_at = 当前时间
   └─ 发送最终进度
      └─ manager.send_progress(task_id, final_progress)
   ↓
返回结果
   ├─ task_id: 任务ID
   ├─ batch_id: 批次ID
   ├─ status: "accepted"
   └─ message: "扫描任务已创建，正在后台执行"
   ↓
前端接收响应
   ├─ 保存task_id和batch_id
   ├─ 连接WebSocket: ws://host/api/ws/scan/{task_id}
   └─ 开始监听进度更新
```

### 2. WebSocket实时进度监控流程

```
前端: connectScanProgress(taskId)
   ↓
检查是否有已有连接
   ├─ 已连接 → 关闭旧连接
   └─ 无连接 → 继续
   ↓
创建WebSocket连接
   ├─ URL: ws://host:port/api/ws/scan/{task_id}
   │  ├─ host: 后端服务器主机名
   │  └─ port: API端口（默认8000，可配置VITE_API_PORT）
   ├─ 使用WebSocketClient工具类
   └─ 配置事件处理器
      ├─ on('message') - 接收进度更新
      ├─ on('connected') - 连接成功
      ├─ on('disconnected') - 连接断开
      └─ on('error') - 连接错误
   ↓
后端: WebSocket连接
   ├─ 接受WebSocket连接
   ├─ manager.connect(websocket, task_id)
   │  ├─ 存储连接: active_connections[task_id].append(websocket)
   │  ├─ 关联任务: connection_tasks[websocket] = task_id
   │  └─ 记录日志
   └─ 连接成功
   ↓
【扫描器推送进度】
   ↓
扫描器每分钟调用: _update_progress(current_file)
   ↓
准备进度数据
   ├─ batch_id
   ├─ task_id
   ├─ status (running/completed/stopped)
   ├─ total_files
   ├─ scanned_files
   ├─ new_files
   ├─ updated_files
   ├─ skipped_files
   ├─ failed_files
   ├─ current_file
   └─ progress (百分比)
   ↓
检查是否有活跃连接
   ├─ 无活跃连接 → 跳过推送，节省资源
   └─ 有活跃连接 → 继续推送
   ↓
通过WebSocket推送
   ├─ manager.send_progress(task_id, progress_data)
   ├─ 遍历active_connections[task_id]的所有连接
   ├─ 对每个连接发送: connection.send_json(progress_data)
   └─ 处理发送失败
      ├─ 捕获异常
      ├─ 记录错误日志
      └─ 清理断开的连接
         ├─ 从active_connections移除
         └─ 从connection_tasks移除
   ↓
【前端接收进度】
   ↓
WebSocketClient触发'onmessage'事件
   ↓
前端处理进度数据
   ├─ 更新scanProgress Map: scanProgress.set(taskId, data)
   ├─ 更新任务列表中的任务
   │  ├─ 查找task: taskList.find(t => t.id === taskId)
   │  └─ 更新任务属性
   │     ├─ task.progress = data.progress
   │     ├─ task.scannedFiles = data.scanned_files
   │     ├─ task.totalFiles = data.total_files
   │     ├─ task.newFiles = data.new_files
   │     ├─ task.updatedFiles = data.updated_files
   │     ├─ task.skippedFiles = data.skipped_files
   │     ├─ task.failedFiles = data.failed_files
   │     ├─ task.currentFile = data.current_file
   │     └─ task.status = data.status
   └─ 更新UI显示
      ├─ 进度条显示
      ├─ 文件数量统计
      ├─ 当前文件显示
      └─ 状态标签更新
   ↓
【扫描完成或停止】
   ↓
最终进度推送
   ├─ status = "completed" 或 "stopped"
   ├─ progress = 100
   └─ current_file = null
   ↓
前端完成处理
   ├─ 更新任务状态为完成
   ├─ 断开WebSocket连接
   └─ 刷新任务列表（可选）
```

### 4. 扫描路径管理流程

#### 4.1 添加扫描路径

```
用户操作
   ↓
点击"添加路径"按钮
   ↓
前端: handleAddPath()
   ↓
打开路径配置对话框
   ↓
用户输入路径信息
   ├─ 路径名称 (path_name)
   │  ├─ 必填，1-100字符
   │  ├─ 用于识别和显示
   │  └─ 验证非法字符
   ├─ 扫描路径 (path)
   │  ├─ 必填，最大260字符
   │  ├─ 支持浏览功能
   │  └─ 验证非法字符和保留设备名
   ├─ 扫描类型 (scan_type)
   │  ├─ full: 全量扫描
   │  └─ incremental: 增量扫描
   ├─ 递归扫描 (recursive)
   ├─ 启用监控 (monitoring_enabled)
   ├─ 监控防抖 (monitoring_debounce)
   │  └─ 单位：秒，范围1-60
   ├─ 忽略模式 (ignore_mode)
   │  ├─ none: 不忽略
   │  ├─ temp: 忽略临时文件
   │  ├─ system: 忽略系统文件
   │  ├─ both: 忽略临时和系统文件
   │  └─ custom: 自定义
   │     └─ 显示多行文本框输入自定义模式
   ├─ 自动扫描间隔 (scan_interval)
```

## 五、状态定义

### 扫描任务状态（ScanHistory）

- `running`: 扫描中
  - 任务正在执行
  - completed_at 为 null
  
- `completed`: 已完成
  - 任务正常完成
  - completed_at 不为 null
  
- `stopped`: 已停止
  - 任务被用户停止
  - completed_at 不为 null
  - error_message 可能包含停止原因
  
- `failed`: 失败
  - 任务执行失败
  - completed_at 不为 null
  - error_message 包含错误信息

### 扫描进度状态（ScanProgress）

- `pending`: 等待中
  - 任务已创建，等待执行
  
- `running`: 运行中
  - 任务正在执行
  
- `stopping`: 停止中
  - 收到停止请求，正在停止
  
- `stopped`: 已停止
  - 任务被停止
  
- `completed`: 已完成
  - 任务正常完成

### 文件任务状态（FileTask）

**扫描阶段**：
- `pending`: 等待中
  - 文件任务已创建，等待处理
  
- `scanning`: 扫描中
  - 正在提取文件元数据
  
- `scanned`: 已扫描
  - 文件元数据已提取完成
  
- `skipped`: 已跳过
  - 文件被跳过（根据skip_mode或ignore_patterns）
  
- `failed`: 扫描失败
  - 文件扫描失败
  - scan_error 包含错误信息

**识别阶段**：
- `recognizing`: 识别中
  - 正在进行网络识别
  
- `recognized`: 已识别
  - 识别成功完成
  
- `recognition_failed`: 识别失败
  - 识别失败
  - recognition_error 包含错误信息

**整理阶段**：
- `organizing`: 整理中
  - 正在进行文件整理
  
- `organized`: 已整理
  - 整理成功完成
  
- `organize_failed`: 整理失败
  - 整理失败
  - organize_error 包含错误信息

## 六、配置说明

### 扫描类型（scan_type）

- `full`: 全量扫描
  - 扫描所有文件
  - 强制更新文件元数据
  - 更新scanned_at时间
  - 适用场景：首次扫描、完整更新

- `incremental`: 增量扫描
  - 仅扫描新文件和已更新的文件
  - 比较修改时间和文件大小
  - 跳过未变化的文件
  - 适用场景：定期更新

- `rescan`: 重新扫描
  - 强制更新指定的文件
  - 不进行跳过检查（除非配置了skip_mode）
  - 适用场景：错误恢复、规则更新后

- `custom`: 自定义路径扫描
  - 使用用户临时指定的路径和配置
  - 适用场景：临时扫描、测试扫描

### 文件跳过模式（skip_mode）

- `keyword`: 仅跳过关键词库文件
  - 跳过文件名包含关键词的文件
  - 扫描所有其他文件
  - 适用场景：更新扫描时间

- `record`: 跳过关键词库和已扫描文件
  - 跳过关键词库文件
  - 跳过数据库中已存在的文件
  - 仅处理新文件
  - 适用场景：增量扫描

- `none`: 不跳过任何文件
  - 强制处理所有文件
  - 适用场景：重新扫描

### 跳过优先级

```
优先级1: 用户忽略模式 (ignore_patterns)
   ↓ 匹配 → 跳过
   ↓ 不匹配 → 继续
   ↓
优先级2: 关键词库文件检查
   ↓ 是关键词库 → 跳过
   ↓ 不是关键词库 → 继续
   ↓
优先级3: 已扫描文件检查 (仅skip_mode=record)
   ↓ 已存在 → 跳过
   ↓ 不存在 → 继续
   ↓
处理文件
```

## 八、关键特性说明

### 1. 批次管理

每次扫描都会生成唯一的批次ID（batch_id），用于：
- 标识一次完整的扫描过程
- 关联扫描历史和文件任务
- 支持按批次查询文件任务
- 区分不同时间的扫描结果

### 2. 文件级别任务跟踪

每个被扫描的文件都有对应的FileTask记录，用于：
- 跟踪文件在扫描、识别、整理各阶段的状态
- 记录每个阶段的开始和完成时间
- 记录各阶段的错误信息
- 支持单独查询和管理每个文件

### 3. 实时进度推送

- 扫描器每分钟更新一次进度
- 通过WebSocket实时推送到前端
- 支持多任务并发监控
- 自动清理断开的连接

### 4. 停止机制

采用"等待实际停止"机制：
1. 用户点击停止按钮
2. API标记任务为"stopping"
3. API等待扫描器实际停止（最多60秒）
4. 扫描器检查停止标志，完成当前文件后终止
5. API确认扫描器已停止后返回结果

### 5. 数据一致性保证

- 所有数据库操作使用事务
- 扫描失败时不影响已扫描的文件
- 停止扫描时保存当前进度
- 支持重新扫描继续处理
