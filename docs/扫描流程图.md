## 一、系统架构概述

### 扫描路径管理模式
系统支持两种扫描路径管理模式：
1. **已配置路径扫描**：使用预先配置的扫描路径及其独立策略
2. **自定义路径扫描**：临时扫描其他目录，使用系统默认配置

### 扫描路径独立配置
每个扫描路径可以独立配置：
- 路径名称（便于识别）
- 扫描路径（必填）
- 扫描类型（full/incremental）
- 递归扫描开关
- 扫描间隔（秒）
- 监控开关
- 监控防抖延迟（秒，范围1-60）
- 忽略文件模式列表（JSON数组）
- 启用状态

### 默认扫描策略配置
当所选路径不在已配置路径中且用户未在手动扫描中选择任何配置时，使用默认扫描策略扫描指定的目录。

**注意**：当前版本中，默认扫描策略配置功能后端API接口尚未实现，前端扫描配置按钮显示"功能开发中"。

**默认扫描策略配置项**（计划实现）：
- `default_scan_type`: 默认扫描类型
  - 默认值：`full`（全量扫描）
  - 可选值：`full`、`incremental`
  - 说明：自定义路径扫描时使用的默认扫描类型

- `default_recursive`: 默认递归扫描开关
  - 默认值：`true`
  - 说明：自定义路径扫描时是否递归扫描子目录

- `default_skip_mode`: 默认文件跳过模式
  - 默认值：`keyword`（仅跳过关键词库文件）
  - 可选值：
    - `none`: 不跳过任何文件
    - `keyword`: 仅跳过关键词库文件
    - `record`: 跳过关键词库和已扫描文件
  - 说明：自定义路径扫描时使用的默认文件跳过模式

- `default_ignore_patterns`: 默认忽略文件模式列表
  - 默认值：`[]`（空列表）
  - 类型：数组
  - 说明：自定义路径扫描时使用的默认忽略文件模式

**默认扫描策略使用场景**：
1. 自定义路径扫描：用户手动输入路径进行扫描
2. 临时扫描：需要快速扫描某个目录而不想配置路径
3. 测试扫描：测试扫描功能而不影响已配置路径

**默认扫描策略配置面板**（计划实现）：
- 位置：扫描管理 → 扫描配置按钮
- 功能：
  - 配置默认扫描策略
  - 保存配置到系统配置文件
  - 重置为系统默认值
- 状态：当前版本中，该功能后端API接口尚未实现，前端显示"功能开发中"

### 文件跳过方式
系统通过`skip_mode`参数控制文件跳过行为，支持三种模式：

1. **keyword模式**（默认）
   - 仅跳过关键词库文件
   - 扫描所有其他文件
   - 适用于需要更新文件扫描时间的场景
   - 跳过文件名包含以下关键词的文件：
     - keywords, keyword_, key_
     - library, lib_
     - rules, rule_
     - mapping, map_
     - season, episode
     - naming, name_

2. **record模式**
   - 跳过关键词库文件
   - 跳过数据库中已存在的文件（基于文件路径判断）
   - 仅处理新文件
   - 适用于增量扫描场景
   - 在scan_type为incremental时使用

3. **none模式**
   - 不跳过任何文件
   - 强制处理所有文件
   - 适用于需要重新扫描所有文件的场景
   - 重新扫描时使用此模式

#### skip_mode参数说明
- **keyword**: 仅跳过关键词库文件
- **record**: 跳过关键词库和已扫描文件
- **none**: 不跳过任何文件

#### 实现逻辑
```python
# 检查用户配置的忽略模式（优先级最高）
if file matches ignore_patterns:
    skip file

# 检查是否为关键词库文件
if _is_keyword_library_file(file_path):
    skip file

# skip_mode为"record"时，跳过已扫描的文件
if skip_mode == 'record' and existing_file and scan_type != 'rescan':
    skip file
```

#### 跳过优先级
```
文件扫描
   ↓
检查用户忽略模式（优先级1）
   ├─ 匹配 → 跳过
   └─ 不匹配 → 继续
   ↓
检查关键词库文件（优先级2）
   ├─ 是 → 跳过
   └─ 否 → 继续
   ↓
检查已扫描文件（优先级3，仅skip_mode=record）
   ├─ 已存在 → 跳过
   └─ 不存在 → 继续
   ↓
处理文件
```

#### WebSocket连接管理策略

**前端策略**：
1. 仅在进度监控组件显示时建立WebSocket连接
2. 组件隐藏或页面离开时主动断开连接
3. 避免不必要的连接和资源消耗

**后端策略**：
1. 检查是否有活跃的WebSocket连接
2. 仅在有活跃连接时推送进度更新
3. 避免向断开的连接推送数据
4. 自动清理断开的连接

#### 连接生命周期

```
组件挂载
   ↓
检查是否有活跃任务
   ├─ 有任务 → 建立WebSocket连接
   └─ 无任务 → 等待
   ↓
建立连接
   ├─ 后端接受连接
   ├─ 存储到连接管理器
   └─ 标记为活跃连接
   ↓
接收进度更新
   ├─ 组件可见 → 更新UI
   └─ 组件不可见 → 忽略
   ↓
组件卸载/隐藏
   ├─ 发送关闭消息
   ├─ 断开WebSocket连接
   └─ 后端清理连接
   ↓
后端检查连接
   ├─ 有活跃连接 → 继续推送
   └─ 无活跃连接 → 停止推送
```

#### 扫描路径配置项说明

**基础配置**：
- `path`: 扫描路径（必填）
- `path_name`: 路径名称（可选，便于识别）
- `enabled`: 是否启用该路径（默认true）

**扫描策略配置**：
- `scan_type`: 扫描类型
  - `full`: 全量扫描（扫描所有文件）
  - `incremental`: 增量扫描（仅扫描新文件和已更新的文件）
- `recursive`: 是否递归扫描子目录
- `scan_interval`: 自动扫描间隔（秒）
  - 0: 不自动扫描
  - >0: 每隔指定秒数自动触发扫描

**监控配置**：
- `monitoring_enabled`: 是否启用文件系统监控
  - 启用: 监控文件变化，自动触发扫描
  - 禁用: 不监控文件变化
- `monitoring_debounce`: 监控防抖延迟（秒，范围1-60）
  - 文件变化后等待指定时间再触发扫描
  - 避免频繁触发扫描

**文件跳过配置**：
- `ignore_patterns`: 忽略文件模式列表
  - 支持通配符模式（如：*.tmp, *.part）
  - 多个模式用逗号分隔
  - 优先级最高，匹配的文件始终被跳过

#### 扫描策略执行流程

```
扫描路径触发
   ↓
检查路径是否启用
   ├─ 未启用 → 跳过
   └─ 已启用 → 继续
   ↓
检查扫描类型
   ├─ 全量扫描 → 使用scan_type=full
   └─ 增量扫描 → 使用scan_type=incremental
   ↓
检查跳过模式
   ├─ 用户忽略模式 → 跳过匹配文件
   ├─ 关键词库文件 → 跳过关键词库文件
   └─ 已扫描文件 → 根据scan_type决定
      ├─ full → 不跳过
      └─ incremental → 跳过未变化的文件
   ↓
执行扫描
   ├─ 遍历目录（根据recursive）
   ├─ 处理媒体文件
   └─ 处理字幕文件
   ↓
更新扫描统计
   ├─ total_scans + 1
   ├─ last_scan_at = 当前时间
   └─ last_scan_batch_id = 当前批次ID
```

```

#### 监控触发机制

```
文件系统监控启动
   ↓
为每个监控路径设置观察者
   ├─ monitoring_enabled = true → 设置观察者
   │  ├─ 监控文件创建事件
   │  ├─ 监控文件修改事件
   │  ├─ 监控文件删除事件
   │  ├─ 监控文件移动事件
   │  └─ 监控目录变化事件
   └─ monitoring_enabled = false → 不设置观察者
   ↓
检测到文件变化
   ├─ 创建事件 → 记录新文件
   ├─ 修改事件 → 记录修改文件
   ├─ 删除事件 → 记录删除文件
   ├─ 移动事件 → 记录移动文件
   └─ 目录变化 → 重新扫描目录
   ↓
应用防抖延迟
   ├─ 等待monitoring_debounce秒
   ├─ 如果有新变化，重置计时器
   └─ 防抖期间的变化会被合并
   ↓
防抖延迟到期
   ↓
检查路径是否启用
   ├─ 未启用 → 跳过扫描
   └─ 已启用 → 继续扫描
   ↓
触发扫描
   ├─ 使用路径的scan_type
   │  ├─ full → 全量扫描
   │  └─ incremental → 增量扫描
   ├─ 使用路径的recursive
   │  ├─ true → 递归扫描子目录
   │  └─ false → 仅扫描当前目录
   └─ 使用路径的ignore_patterns
      ├─ 匹配的文件被跳过
      └─ 未匹配的文件被处理
   ↓
更新扫描统计
   ├─ total_scans + 1
   ├─ last_scan_at = 当前时间
   └─ last_scan_batch_id = 当前批次ID
```

#### 监控事件处理

**文件创建事件**：
- 记录新创建的文件路径
- 在防抖延迟到期后触发扫描
- 扫描时会处理新创建的文件

**文件修改事件**：
- 记录修改的文件路径
- 在防抖延迟到期后触发扫描
- 增量扫描时检查文件是否真的被修改

**文件删除事件**：
- 记录删除的文件路径
- 在防抖延迟到期后触发扫描
- 从数据库中删除对应的文件记录

**文件移动事件**：
- 记录源路径和目标路径
- 在防抖延迟到期后触发扫描
- 更新数据库中的文件路径

**目录变化事件**：
- 记录变化的目录路径
- 在防抖延迟到期后触发扫描
- 重新扫描整个目录

#### 监控防抖机制

**防抖作用**：
- 避免频繁触发扫描
- 合并短时间内的多个变化
- 减少系统资源消耗

**防抖流程**：
```
文件变化事件
   ↓
启动防抖计时器
   ├─ 等待monitoring_debounce秒
   └─ 如果有新变化，重置计时器
   ↓
防抖计时器到期
   ↓
检查是否有待处理的变化
   ├─ 有变化 → 触发扫描
   └─ 无变化 → 结束
   ↓
执行扫描
   ↓
清空变化记录
```

**防抖配置建议**：
- 小型目录：5秒
- 中型目录：10秒
- 大型目录：30秒
- 高频变化目录：60秒

**配置说明**：
- 最小单位：秒
- 配置范围：1-60秒
- 根据目录大小和变化频率选择合适的值

#### 5.2 重新扫描类型

**1. 重新扫描所有文件**
- 重新扫描原始任务中的所有文件
- 强制更新所有文件的元数据
- 根据重新扫描选项决定是否跳过文件
  - skip_keywords: 控制是否跳过关键词库文件（默认true）
  - skip_scanned: 控制是否跳过已扫描文件（默认false）
  - use_ignore_patterns: 控制是否使用忽略模式（默认true）
- 适用场景：
  - 文件识别规则更新后
  - 需要重新提取文件信息
  - 系统升级后需要重新识别

**2. 仅重新扫描失败的文件**
- 仅重新扫描原始任务中失败的文件
- 跳过成功和跳过的文件
- 根据重新扫描选项决定是否跳过文件
  - skip_keywords: 控制是否跳过关键词库文件（默认true）
  - skip_scanned: 控制是否跳过已扫描文件（默认true）
  - use_ignore_patterns: 控制是否使用忽略模式（默认true）
- 适用场景：
  - 临时网络问题导致失败
  - 文件锁定问题已解决
  - 系统资源问题已恢复

**3. 仅重新扫描指定的文件**
- 仅重新扫描用户指定的文件
- 跳过其他文件
- 根据重新扫描选项决定是否跳过文件
  - skip_keywords: 控制是否跳过关键词库文件（默认true）
  - skip_scanned: 控制是否跳过已扫描文件（默认false）
  - use_ignore_patterns: 控制是否使用忽略模式（默认true）
- 适用场景：
  - 某些文件识别结果不正确
  - 需要更新特定文件的信息
  - 文件内容已更新需要重新识别

#### 5.3 重新扫描选项

**配置选项**：
- `rescan_type`: 重新扫描类型
  - `all`: 重新扫描所有文件
  - `failed`: 仅重新扫描失败的文件
  - `selected`: 仅重新扫描指定的文件

- `file_list`: 指定文件列表（仅rescan_type=selected时使用）
  - 类型：数组
  - 内容：文件路径列表

- `force_update`: 是否强制更新
  - 默认值：true
  - 说明：是否强制更新文件元数据

- `skip_validation`: 是否跳过验证
  - 默认值：false
  - 说明：是否跳过文件验证

- `skip_keywords`: 是否跳过关键词库文件
  - 默认值：true
  - 说明：是否跳过关键词库文件
  - 适用场景：
    - true: 跳过关键词库文件，仅扫描媒体文件
    - false: 不跳过关键词库文件，重新扫描所有文件

- `skip_scanned`: 是否跳过已扫描文件
  - 默认值：false
  - 说明：是否跳过已扫描的文件
  - 适用场景：
    - true: 跳过已扫描文件，仅扫描新文件
    - false: 不跳过已扫描文件，重新扫描所有文件

- `use_ignore_patterns`: 是否使用忽略模式
  - 默认值：true
  - 说明：是否使用用户配置的忽略模式
  - 适用场景：
    - true: 使用忽略模式，跳过匹配的文件
    - false: 不使用忽略模式，扫描所有文件

#### 5.4 重新扫描注意事项

**注意事项**：
1. 重新扫描会更新所有文件的扫描时间
2. 重新扫描不计入新增文件统计
3. 重新扫描会创建新的扫描历史记录
4. 重新扫描会保留原始任务记录
5. 重新扫描不会影响其他任务的文件记录
6. 重新扫描会使用原始任务的配置
7. 重新扫描根据选项决定是否使用文件跳过模式
   - skip_keywords: 控制是否跳过关键词库文件（默认true）
   - skip_scanned: 控制是否跳过已扫描文件（默认false）
   - use_ignore_patterns: 控制是否使用忽略模式（默认true）
8. 重新扫描会强制更新文件元数据

**性能考虑**：
1. 重新扫描所有文件可能耗时较长
2. 建议在系统空闲时进行
3. 大量文件重新扫描可能影响系统性能
4. 可以选择仅重新扫描部分文件以减少影响

**数据一致性**：
1. 重新扫描会更新文件的扫描时间
2. 重新扫描会更新文件的识别结果
3. 重新扫描不会删除原始扫描记录
4. 重新扫描会创建新的扫描历史

### 6. 停止扫描任务流程

#### 6.1 停止扫描任务流程

   ↓

   ├─ 查询数据库获取任务状态
   ├─ 检查扫描器是否已停止

   ├─ 完成当前文件处理

   └─ 更新任务状态为"stopped"

   ↓
【扫描器已完全停止】
   ↓

   ├─ 显示停止成功提示
   ├─ 显示停止耗时
   └─ 刷新任务列表
```

#### 6.2 停止机制说明

**停止原理**：
1. 扫描任务在后台异步执行
2. 停止操作通过更新数据库中的任务状态实现
3. 扫描器在处理每个文件时会检查任务状态
4. 如果任务已被标记为停止，扫描器会终止扫描
5. API等待扫描器实际停止后再返回结果

**重要说明 - 等待实际停止的机制**：

为解决Windows文件读取操作不会立即停止的问题，采用"等待实际停止"机制：

1. **停止请求阶段**
   - 用户点击停止按钮
   - API接收停止请求
   - 数据库更新任务状态为"stopping"
   - 记录停止请求时间
   - **时间：通常< 100ms**

2. **等待停止阶段**
   - API不立即返回结果
   - 循环检查任务状态
   - 等待扫描器实际停止
   - 每秒检查一次
   - **时间：取决于文件处理进度，通常1-30秒**

3. **扫描器停止阶段**
   - 扫描器检查到停止标志
   - 完成当前文件处理
   - 终止扫描循环
   - 更新任务状态为"stopped"
   - **时间：取决于文件大小，可能需要几秒到几十秒**

4. **返回结果阶段**
   - 确认扫描器已完全停止
   - 返回停止成功结果
   - 包含实际停止时间
   - 包含停止耗时
   - **时间：通常< 100ms**

**优势**：
- 用户看到"停止成功"时，扫描已真正停止
- 避免用户误解和重复操作
- 提供准确的停止耗时信息
- 确保数据一致性

**时间差的影响因素**：

1. **文件大小**
   - 小文件（< 10MB）：通常1-5秒
   - 中等文件（10-100MB）：通常5-15秒
   - 大文件（> 100MB）：可能15-60秒或更长

2. **文件类型**
   - 媒体文件：需要读取文件头信息，时间较长
   - 字幕文件：文件较小，时间较短
   - 其他文件：取决于文件大小

3. **系统负载**
   - CPU使用率高：文件处理变慢
   - 磁盘I/O繁忙：文件读取变慢
   - 内存不足：影响处理速度

4. **网络存储**
   - 本地存储：读取速度较快
   - 网络驱动器：读取速度较慢
   - 云存储：受网络带宽影响

**停止流程（等待实际停止）**：
```
用户点击停止
   ↓
API标记任务为停止中
   ├─ API接收请求
   ├─ 数据库更新任务状态
   │  ├─ status = "stopping"
   │  ├─ stop_requested_at = 当前时间
   │  └─ stop_reason = "用户请求停止"
   └─ 提交数据库事务
   ↓
【等待扫描器实际停止】
   ↓
循环检查任务状态
   ├─ 查询数据库获取任务状态
   │  ├─ status = "stopped" → 已停止，退出循环
   │  └─ status = "stopping" → 继续等待
   ├─ 检查是否超时
   │  ├─ 超时时间：60秒
   │  ├─ 未超时 → 继续等待
   │  └─ 超时 → 返回部分成功
   └─ 等待间隔：1秒
      └─ 使用time.sleep(1)避免频繁查询
   ↓
扫描器停止文件读取
   ├─ 完成当前文件处理
   │  ├─ Windows文件读取操作无法立即停止
   │  ├─ 需要等待当前文件读取完成
   │  └─ 时间：取决于文件大小和系统负载
   ├─ 检查到停止标志
   │  ├─ 在_update_progress时检查
   │  ├─ 在处理文件前检查
   │  └─ 在更新数据库前检查
   ├─ 终止扫描循环
   │  ├─ 不再处理下一个文件
   │  └─ 跳出扫描循环
   └─ 更新任务状态为"stopped"
      ├─ status = "stopped"
      ├─ completed_at = 当前时间
      └─ error_message = "任务被用户停止"
   ↓
【扫描器已完全停止】
   ↓
保存当前扫描结果
   ├─ 保存已扫描的文件
   ├─ 更新扫描进度为停止状态
   └─ 记录停止原因
   ↓
返回停止结果
   ├─ success: true
   ├─ message: "扫描任务已停止"
   ├─ stop_time: 实际停止时间
   └─ stop_duration: 停止耗时（秒）
```

**状态检查点**：
1. **初始化阶段**
   - 开始扫描前检查任务状态
   - 如果已停止，直接返回

2. **文件处理阶段**
   - 处理每个文件前检查任务状态
   - 如果已停止，跳出循环

3. **进度更新阶段**
   - 更新进度时检查任务状态
   - 如果已停止，不再继续处理

4. **数据库操作阶段**
   - 更新数据库前检查任务状态
   - 如果已停止，保存当前结果

**停止后的处理**：
1. 保存已扫描的文件记录
2. 更新扫描进度为停止状态
3. 记录停止原因和停止时间
4. 通过WebSocket通知前端任务已停止
5. 释放扫描器占用的资源

#### 6.3 停止注意事项

**注意事项**：
1. 停止操作不可逆，一旦停止无法继续
2. 停止后会保存当前扫描结果
3. 已扫描的文件不会被删除
4. 未扫描的文件需要重新触发扫描
5. 停止操作不会影响其他运行中的任务
6. 已完成的任务无法停止

**性能考虑**：
1. 停止操作采用等待实际停止机制
2. API响应时间取决于扫描器实际停止时间
3. 需要等待当前文件处理完成
4. 建议在文件数较少时停止
5. 大量文件扫描时停止可能需要较长时间

**等待实际停止的性能影响**：

1. **API响应时间**
   - API响应时间 = 扫描器实际停止时间
   - 通常1-30秒，极端情况可能更长
   - 用户需要等待API响应
   - 前端应显示"正在停止..."状态

2. **数据库查询频率**
   - 每秒查询一次任务状态
   - 避免频繁查询影响性能
   - 使用time.sleep(1)控制查询间隔
   - 超时时间：60秒

3. **并发停止请求**
   - 同一任务的重复停止请求
   - 检查任务状态是否为"stopping"
   - 如果正在停止，返回"正在停止中"
   - 避免重复处理

4. **超时处理**
   - 如果60秒内未停止，返回部分成功
   - 记录超时事件
   - 通知管理员处理
   - 提供强制停止选项

5. **资源占用**
   - 等待期间占用HTTP连接
   - 建议使用异步API
   - 考虑使用WebSocket推送停止完成事件
   - 优化用户体验

**优势**：
- 用户看到"停止成功"时，扫描已真正停止
- 避免用户误解和重复操作
- 提供准确的停止耗时信息
- 确保数据一致性

**数据一致性**：
1. 停止后会保存已扫描的文件
2. 扫描进度会更新为停止状态
3. 停止原因会被记录
4. 可以通过重新扫描继续扫描剩余文件

## 七、配置参数

### 扫描配置
- recursive: 是否递归扫描子目录
- scan_type: 扫描类型 (full/incremental/rescan)
- interval: 定时扫描间隔（秒），范围：300-600秒

### 默认扫描策略配置
**注意**：当前版本中，默认扫描策略配置功能后端API接口尚未实现，以下为计划实现的配置项。

**配置项**（计划实现）：
- `default_scan_type`: 默认扫描类型
  - 默认值：`full`
  - 可选值：`full`、`incremental`
  - 说明：自定义路径扫描时使用的默认扫描类型

- `default_recursive`: 默认递归扫描开关
  - 默认值：`true`
  - 说明：自定义路径扫描时是否递归扫描子目录

- `default_skip_mode`: 默认文件跳过模式
  - 默认值：`keyword`
  - 可选值：
    - `none`: 不跳过任何文件
    - `keyword`: 仅跳过关键词库文件
    - `record`: 跳过关键词库和已扫描文件
  - 说明：自定义路径扫描时使用的默认文件跳过模式

- `default_ignore_patterns`: 默认忽略文件模式列表
  - 默认值：`[]`
  - 类型：数组
  - 说明：自定义路径扫描时使用的默认忽略文件模式

**配置管理**（计划实现）：
- 配置文件位置：`config/app.yaml`
- 配置修改后需要重启服务生效
- 支持通过API动态更新配置

**API接口**（计划实现）：
- 获取默认扫描策略：`GET /api/scan/config/default`
- 更新默认扫描策略：`PUT /api/scan/config/default`
- 重置默认扫描策略：`POST /api/scan/config/default/reset`

### 监控配置
- enabled: 是否启用文件系统监控
  - true: 启用实时监控
  - false: 禁用监控，仅使用定时扫描
- debounce_seconds: 防抖延迟时间（秒）
  - 范围：1-60秒
  - 默认值：5秒
  - 作用：文件变化后等待指定时间再触发扫描
- recursive: 是否递归监控子目录
- ignore_patterns: 忽略文件模式列表
  - 默认值：`["*.tmp", "*.part", ".*"]`
  - 说明：匹配这些模式的文件将被忽略
