**更新日期**: 2026-03-14

**说明**: 根据新需求重新设计的文件扫描流程（单文件批次）

## 一、系统架构概述

### 核心组件

```
┌─────────────────────────────────────────────────────────────┐
│                      前端界面                            │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │ 扫描路径管理 │  │ 扫描任务管理 │  │ 文件任务管理 │ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
└────────────────────┬────────────────────────────────────┘
                     │ WebSocket / HTTP API
                     ↓
┌─────────────────────────────────────────────────────────────┐
│                      后端服务                            │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐ │
│  │ 文件队列管理 │  │ 文件扫描器  │  │ 进度报告器  │ │
│  └──────────────┘  └──────────────┘  └──────────────┘ │
│         ↓                ↓                ↓              │
│  ┌──────────────────────────────────────────────┐       │
│  │              数据库层                       │       │
│  │  scan_paths | file_tasks | media_files   │       │
│  │  keyword_rules                              │       │
│  └──────────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────────┘
```

### 数据模型关系

```
scan_paths (扫描路径)
   ├─ id
   ├─ path_name
   ├─ path
   ├─ scan_type (扫描类型)
   ├─ recursive (递归方式)
   ├─ skip_strategy (跳过策略)
   ├─ scan_subdirectories (是否扫描子目录)
   ├─ scan_debounce_time (防抖时间)
   ├─ monitoring_enabled (是否监控)
   ├─ monitoring_mode (监控模式)
   ├─ monitoring_debounce (监控防抖)
   ├─ auto_recognize (自动识别)
   ├─ auto_organize (自动整理)
   └─ enabled (是否启用)

file_tasks (文件任务)
   ├─ id (文件任务ID)
   ├─ batch_id (批次ID，每个文件一个批次)
   ├─ media_file_id (媒体文件ID)
   ├─ file_name (文件名)
   ├─ file_path (文件路径)
   ├─ status (状态)
   ├─ scan_started_at (扫描开始)
   ├─ scan_completed_at (扫描完成)
   ├─ scan_progress (扫描进度)
   ├─ video_tracks (视频轨道)
   ├─ audio_tracks (音频轨道)
   ├─ subtitle_tracks (字幕轨道)
   ├─ video_codec (视频编码)
   ├─ has_external_subtitle (外挂字幕)
   ├─ external_subtitle_name (外挂字幕名)
   ├─ scan_result (扫描结果)
   └─ scan_type (扫描类型)

media_files (媒体文件)
   ├─ id
   ├─ file_path
   ├─ file_name
   ├─ file_extension
   ├─ file_size
   ├─ sha256_hash (文件哈希值)
   ├─ video_codec
   ├─ audio_codec
   ├─ duration
   ├─ width
   ├─ height
   ├─ video_tracks
   ├─ audio_tracks
   └─ subtitle_tracks

keyword_rules (关键词规则)
   ├─ id
   ├─ library_id (关键词库ID)
   ├─ pattern (匹配模式)
   ├─ priority (优先级)
   └─ is_enabled (是否启用)
```

## 二、主要流程图

### 2.1 默认扫描策略配置流程

```
用户操作
   ↓
打开系统设置
   ↓
选择"默认扫描策略"
   ↓
系统加载当前策略配置
   ├─ 从配置表读取 scan.default_strategy
   └─ 显示策略参数
      ├─ 扫描类型: 全量/增量
      ├─ 递归方式: 递归/非递归
      ├─ 文件跳过策略: 根据扫描类型显示选项
      ├─ 是否扫描子目录: 是/否
      └─ 扫描任务监控防抖时间: 30-300秒
   ↓
用户修改策略参数
   ↓
【动态更新跳过策略选项】
   ↓
用户选择扫描类型
   ↓
系统动态更新跳过策略选项
   ├─ 如果选择"全量扫描"
   │  └─ 仅显示"不跳过任何文件"选项
   │
   └─ 如果选择"增量扫描"
      └─ 显示三个选项:
         ├─ 仅跳过关键词库文件
         ├─ 跳过关键词库或已扫描文件
         └─ 不跳过任何文件
   ↓
用户点击"保存"
   ↓
前端验证输入
   ├─ 防抖时间: 30-300秒范围验证
   ├─ 必填项检查
   └─ 数据类型验证
   ↓
调用API: PUT /api/scan/default-strategy
   ↓
后端保存策略
   ├─ 更新配置表
   ├─ 序列化为JSON
   └─ 记录配置历史
   ↓
返回保存结果
   ├─ 成功: 200 OK
   └─ 失败: 400 Bad Request
   ↓
前端显示结果
   ├─ 成功提示
   └─ 刷新策略显示
```

### 2.2 添加扫描路径流程

```
用户操作
   ↓
点击"添加路径"按钮
   ↓
打开路径配置对话框
   ↓
【路径基本信息】
   ↓
用户输入路径名称
   ├─ 必填，1-100字符
   └─ 不能包含特殊字符
   ↓
用户选择扫描路径
   ├─ 支持浏览按钮选择
   └─ 支持手动输入
   ↓
【扫描策略配置】
   ↓
配置扫描类型
   ├─ 全量扫描
   └─ 增量扫描
   ↓
配置递归方式
   ├─ 递归扫描
   └─ 非递归扫描
   ↓
配置文件跳过策略
   ├─ 仅跳过关键词库文件
   ├─ 跳过关键词库或已扫描文件
   └─ 不跳过任何文件
   ↓
配置是否扫描子目录
   ├─ 是
   └─ 否
   ↓
配置扫描任务监控防抖时间
   ├─ 默认: 30秒
   └─ 范围: 30-300秒
   ↓
【文件监控配置】
   ↓
配置是否开启文件监控
   ├─ 是
   └─ 否
   ↓
配置文件监控模式
   ├─ watchdog (实时监控)
   └─ polling (轮询监控)
   ↓
配置文件监控防抖时间
   ├─ 默认: 5秒
   └─ 范围: 1-60秒
   ↓
【自动化配置】
   ↓
配置扫描完成后是否自动识别
   ├─ 是
   └─ 否
   ↓
配置扫描完成后是否自动整理
   ├─ 是
   └─ 否
   ↓
用户点击"确定"
   ↓
前端验证输入
   ├─ 路径名称: 非空、长度、特殊字符
   ├─ 扫描路径: 非空、路径有效性
   ├─ 路径唯一性: 检查是否已存在
   └─ 防抖时间: 范围验证
   ↓
调用API: POST /api/scan/paths
   ↓
后端验证和处理
   ├─ 检查路径是否存在
   │  └─ 不存在 → 返回404
   ├─ 检查路径是否为目录
   │  └─ 不是目录 → 返回400
   ├─ 检查路径是否已配置
   │  └─ 已存在 → 返回409
   └─ 创建扫描路径记录
      ├─ 保存到 scan_paths 表
      ├─ 初始化统计字段
      │  ├─ total_scans = 0
      │  ├─ total_files_found = 0
      │  ├─ last_scan_at = null
      │  └─ last_scan_batch_id = null
      └─ 记录创建时间
   ↓
返回创建结果
   ├─ 成功: 返回路径ID和基本信息
   └─ 失败: 返回错误信息
   ↓
前端显示结果
   ├─ 成功提示
   └─ 刷新路径列表
```

### 2.3 手动触发扫描流程（路径列表）

```
用户操作
   ↓
在路径列表中选中一个或多个路径
   ↓
点击"扫描"按钮
   ↓
弹出扫描选择对话框
   ↓
显示选中路径信息
   ├─ 路径名称列表
   ├─ 扫描路径列表
   └─ 当前策略摘要
   ↓
用户选择扫描策略
   ├─ 使用默认扫描策略
   │  ├─ 读取全局默认策略
   │  └─ 应用到所有选中路径
   │
   └─ 使用路径对应的扫描策略
      ├─ 使用每个路径自己的策略
      └─ 策略可能不同
   ↓
用户确认选择
   ↓
前端构建扫描请求
   ├─ path_ids: [id1, id2, ...]
   └─ use_default_strategy: true/false
   ↓
调用API: POST /api/scan/tasks
   ↓
后端处理请求
   ├─ 验证路径ID有效性
   ├─ 读取路径配置
   ├─ 确定使用的策略
   │  └─ use_default_strategy = true
   │     ├─ 读取默认策略
   │     └─ 应用默认策略
   │
   │  └─ use_default_strategy = false
   │     └─ 使用路径自身策略
   │
   └─ 遍历路径，收集文件
      ├─ 根据 recursive 配置遍历
      ├─ 对每个文件执行类型判定
      │  ├─ 检查文件扩展名
      │  └─ 检查编码格式
      │  └─ 两者都符合 → 加入文件队列
      │
      └─ 不符合 → 跳过
   ↓
【文件队列管理】
   ↓
FileQueueManager 接收文件列表
   ├─ 为每个文件创建独立的批次
   ├─ 生成唯一的 batch_id (每个文件一个)
   ├─ 创建 file_tasks 记录
   ├─ status = "pending"
   └─ 添加到文件队列
   ↓
启动文件扫描
   ├─ 一次只处理一个文件
   ├─ 串行处理，不并发
   └─ 当前文件完成后处理下一个
   ↓
返回任务信息
   ├─ task_ids: [123, 124, 125] (每个文件一个任务ID)
   ├─ status: "accepted"
   └─ message: "扫描任务已创建"
   ↓
前端显示任务
   ├─ 添加到文件任务列表
   ├─ 连接WebSocket监听进度
   └─ 实时更新进度显示
```

### 2.5 文件类型双重判定流程

```
扫描器发现文件
   ↓
【第一步: 检查文件扩展名】
   ↓
提取文件扩展名
   ├─ 获取文件名后缀
   └─ 转换为小写
   ↓
检查扩展名是否在支持列表中
   ├─ 支持的扩展名:
   │  ├─ .mp4, .mkv, .avi, .mov
   │  ├─ .rmvb, .ts, .m2ts, .mts
   │  ├─ .wmv, .flv, .webm, .vob
   │  ├─ .ogv, .ogg, .mpg, .mpeg
   │  ├─ .m4v, .3gp, .3g2, .f4v, .f4p
   │  └─ ... (共22种)
   │
   ├─ 扩展名在列表中 → 继续
   └─ 扩展名不在列表中 → 跳过文件
      └─ 不加入扫描队列
   ↓
【第二步: 检查编码格式】
   ↓
使用 pymediainfo 提取媒体信息
   ├─ MediaInfo.parse(file_path)
   ├─ 获取视频轨道
   └─ 提取 codec_id
   ↓
检查是否包含视频轨道
   ├─ 有视频轨道 → 继续
   └─ 无视频轨道 → 跳过文件
   ↓
标准化编码格式
   ├─ 转换为小写
   ├─ 移除空格
   └─ 处理常见别名
   ↓
检查编码格式是否在支持列表中
   ├─ 支持的编码格式:
   │  ├─ mpeg-1, mpeg-2, mpeg-4
   │  ├─ x264, x265
   │  ├─ av1
   │  ├─ vc9
   │  ├─ h264, h265, hevc
   │  └─ vp9
   │
   ├─ 编码格式在列表中 → 加入扫描队列
   │  └─ 创建文件任务
   │     ├─ file_tasks 表
   │     ├─ status = "pending"
   │     └─ 关联 media_file (如果已存在)
   │
   └─ 编码格式不在列表中 → 跳过文件
      └─ 不加入扫描队列
```

### 2.6 文件哈希值计算流程

```
文件通过类型判定
   ↓
准备计算哈希值
   ├─ 初始化 SHA-256 哈希对象
   └─ 创建哈希上下文
   ↓
【分块读取文件】
   ↓
打开文件（二进制模式）
   ↓
循环读取文件块
   ├─ 每次读取 8MB 数据
   ├─ 更新哈希对象
   │  └─ hash.update(chunk)
   ├─ 检查是否到达文件末尾
   │  └─ 未到达 → 继续读取
   └─ 到达 → 退出循环
   ↓
关闭文件
   ↓
获取最终哈希值
   ├─ 转换为十六进制字符串
   └─ 生成 64 字符字符串
   ↓
存储哈希值
   ├─ 保存到 media_files 表
   │  └─ sha256_hash 字段
   └─ 创建唯一索引
      └─ 优化查询性能
   ↓
返回哈希值
   └─ 用于后续文件判定
```

### 2.7 基于哈希值的已扫描文件判定流程

```
扫描器处理文件
   ↓
【计算当前文件哈希值】
   ↓
调用哈希计算函数
   ├─ calculate_file_hash(file_path)
   └─ 获取当前文件的 SHA-256 哈希值
   ↓
检查跳过策略
   ↓
根据 skip_strategy 判定
   ↓
【策略1: keyword】
   ↓
仅检查关键词库
   ├─ 查询关键词库
   │  └─ 匹配文件名
   ├─ 匹配关键词库 → 跳过扫描
   └─ 不匹配 → 继续扫描
   ↓
【策略2: keyword_or_scanned】
   ↓
先检查关键词库
   ├─ 查询关键词库
   ├─ 匹配关键词库 → 跳过扫描
   └─ 不匹配 → 继续检查
   ↓
检查是否已扫描（通过哈希值）
   ↓
查询数据库
   ├─ 查询 media_files 表
   │  └─ WHERE sha256_hash = ?
   │
   ├─ 找到相同哈希值
   │  ├─ 哈希值相同 → 文件已扫描
   │  ├─ 比较文件路径
   │  │  ├─ 路径相同 → 跳过扫描
   │  │  └─ 路径不同 → 重复文件，跳过扫描
   │  └─ 记录为 skipped
   │
   └─ 未找到相同哈希值
      └─ 文件未扫描 → 继续扫描
   ↓
【策略3: none】
   ↓
不执行任何跳过检查
   └─ 直接扫描文件
```

### 2.8 文件队列串行处理流程

```
【添加文件到队列】
   ↓
扫描器收集文件
   ↓
为每个文件创建独立批次
   ├─ 生成唯一的 batch_id (UUID)
   ├─ 创建 file_tasks 记录
   │  ├─ batch_id = 唯一ID
   │  ├─ file_path
   │  ├─ file_name
   │  ├─ status = "pending"
   │  ├─ scan_type
   │  ├─ skip_strategy
   │  └─ 其他字段
   │
   └─ 添加到文件队列
      └─ FileQueueManager.add_file_task(file_path, strategy)
   ↓
【队列串行处理】
   ↓
FileQueueManager 管理队列
   ├─ file_queue = []  # 文件队列
   ├─ current_task = None  # 当前处理的任务
   └─ is_processing = False  # 是否正在处理
   ↓
尝试启动下一个文件
   ↓
检查是否正在处理
   ├─ is_processing = True
   │  └─ 正在处理 → 不启动新任务
   │
   └─ is_processing = False
      └─ 未处理 → 尝试启动
         ↓
         检查队列是否为空
         ├─ 队列为空 → 返回
         └─ 队列不为空 → 继续处理
            ↓
            从队列取出文件
            ├─ task_config = file_queue.pop(0)  # FIFO
            ↓
            标记为正在处理
            ├─ is_processing = True
            └─ current_task = task_id
            ↓
            启动文件扫描
            └─ FileScanner.scan_file(file_path)
   ↓
【处理当前文件】
   ↓
FileScanner 扫描单个文件
   ├─ 检查停止标志
   │  ├─ 已停止 → 退出
   │  └─ 未停止 → 继续
   │
   ├─ 文件类型判定
   │  ├─ 检查扩展名
   │  └─ 检查编码格式
   │
   ├─ 计算文件哈希值
   │  └─ SHA-256 哈希
   │
   ├─ 跳过策略检查
   │  └─ 根据 skip_strategy 判定
   │
   ├─ 提取文件元数据
   │  └─ 使用 pymediainfo
   │
   ├─ 创建/更新 media_file 记录
   │  └─ 存储哈希值
   │
   ├─ 更新 file_tasks 记录
   │  ├─ status = "scanned"
   │  ├─ scan_completed_at = 当前时间
   │  └─ scan_progress = 100
   │
   └─ WebSocket 推送进度
      └─ 实时更新前端
   ↓

   ↓
FileScanner 扫描完成
   ↓
通知 FileQueueManager
   └─ on_task_completed(task_id)

   ├─ is_processing = False
   ├─ current_task = None
   └─ 尝试启动下一个文件
      └─ _try_process_next()
   ↓

   ├─ FIFO (先进先出)
   ├─ 一次只处理一个文件
   ├─ 串行处理，不并发

```

   └─ progress_reporter.report_progress(progress_data)

   │  └─ 立即发送
   │
   └─ 当前时间 - 上次更新时间 < debounce_time
      └─ 等待防抖时间
         └─ 设置定时器
            └─ 防抖时间到后发送
   ↓

   └─ await manager.send_progress(task_id, progress_data)
      ├─ 获取活跃连接
      │  └─ active_connections.get(task_id, [])
      │
      ├─ 遍历所有连接
      │  └─ for connection in connections:
      │     └─ await connection.send_json(progress_data)
      │
      └─ 处理发送失败
         ├─ 捕获异常
         ├─ 记录错误
         └─ 清理断开的连接

   ↓

   ↓
WebSocket接收消息
   ↓
触发 onmessage 事件
   ↓
解析进度数据
   ↓

   └─ scanProgress.set(taskId, progressData)
   ↓
更新UI显示
   ├─ 更新文件列表中的文件
   │  ├─ 查找对应文件
   │  │  └─ task = fileList.find(t => t.id === taskId)

## 六、关键特性说明

### 6.1 文件类型双重判定

**目的**: 确保只扫描真正的视频文件，避免误扫描

**实现**:
1. 第一层：文件扩展名判定
   - 检查文件扩展名是否在支持列表中
   - 支持22种常见视频容器格式

2. 第二层：文件编码格式判定
   - 使用 pymediainfo 提取视频编码信息
   - 检查编码格式是否在支持列表中
   - 支持10种主流视频编码格式

**优势**:
- 避免扫描伪装成视频的非视频文件
- 提高扫描准确性
- 减少无效扫描

### 6.2 基于SHA-256哈希值的文件判定

**目的**: 通过文件哈希值准确判定文件是否已扫描

**实现**:
1. 文件扫描完成后计算SHA-256哈希值
   - 分块读取文件（每次8MB）
   - 避免内存溢出
   - 生成64字符十六进制字符串

2. 存储哈希值到数据库
   - 存储在 media_files 表的 sha256_hash 字段
   - 创建唯一索引，优化查询性能

3. 增量扫描时的文件判定
   - 计算当前文件哈希值
   - 查询数据库中是否存在相同哈希值
   - 哈希值相同 → 文件已扫描，跳过
   - 哈希值不同 → 文件未扫描或已更新，继续扫描

**优势**:
- 基于文件内容而非文件名或时间戳
- 准确检测文件变更
- 避免重复扫描相同文件
- 支持文件去重

### 6.3 文件队列串行处理

**目的**: 避免磁盘IO并发导致的磁盘IO风暴

**实现**:
- 单文件批次：每个文件作为一个独立的批次
- 串行处理：所有文件按队列顺序串行处理
- 一次一文件：一次只处理一个文件
- 不进行并发：不进行磁盘IO并发
- FIFO队列：按先进先出顺序处理

**优势**:
- 避免磁盘IO风暴
- 简化并发控制逻辑
- 提高机械硬盘性能
- 减少系统资源消耗
- 避免磁盘头频繁移动

### 6.4 可配置的防抖机制

**目的**: 避免频繁更新影响性能

**实现**:
- 扫描进度防抖：30-300秒
- 文件监控防抖：1-60秒
- 配置灵活，适应不同场景

**优势**:
- 减少数据库写入
- 降低网络传输
- 提高系统响应速度

### 6.5 单文件批次设计

**设计原则**:
- 每个文件作为一个独立的批次
- 每个文件有独立的任务ID和批次ID
- 不再使用目录级别的批次管理

**优势**:
- 细粒度控制：每个文件独立管理
- 串行处理：避免磁盘IO并发
- 防止IO风暴：单线程处理所有文件
- 简化逻辑：不需要复杂的并发控制
- 精确跟踪：每个文件的状态独立可查

**实现方式**:
- 文件队列：FIFO队列，所有文件按顺序处理
- 一次一文件：当前文件处理完成后，处理下一个
- 批量操作：选中N个文件即创建N个批次
